## Bugs gefunden & gefixt (P1)

### Fix 1: graph_update_node LogRecord Collision
- **Datei:** `mcp_server/tools/graph_update_node.py` Zeilen 92, 107
- **Problem:** `extra={"name": name}` in Logger-Calls kollidiert mit Pythons eingebautem `LogRecord.name` Attribut → Server-Crash
- **Fix:** `"name"` → `"node_name"` in extra-Dicts
- **Verifiziert:** vector_id erfolgreich auf TEST-NODE-AUDIT gesetzt

### Fix 2: graph_find_path missing await (Node Lookup)
- **Datei:** `mcp_server/db/graph.py` Zeilen 1346-1347
- **Problem:** `get_node_by_name()` ist async, wurde ohne `await` aufgerufen → `'coroutine' object is not subscriptable`
- **Fix:** `await` vor beiden Calls
- **Verifiziert:** Keine Crashes mehr bei Node-Lookup

### Fix 3: graph_find_path missing await (Edge Detail)
- **Datei:** `mcp_server/db/graph.py` Zeilen 1486, 1509
- **Problem:** `get_edge_by_id()` ist async, ohne `await` in Relevance-Scoring und IEF-Scoring Loops → `'coroutine' object has no attribute 'get'`
- **Fix:** `await` vor beiden Calls (2 Stellen)
- **Verifiziert:** ethr→Laura (1 Hop), I/O→Drift-Projekt (1 Hop) funktionieren

### Fix 4: graph_find_path BFS Multi-Hop kaputt (SCHWERWIEGENDSTER BUG)
- **Datei:** `mcp_server/db/graph.py` Zeilen 1356-1401
- **Problem:** Die CTE Base Case suchte nur direkte Kanten zwischen Start UND Ziel:
  ```sql
  WHERE (source_id = START AND target_id = END)
     OR (source_id = END AND target_id = START)
  ```
  Wenn kein direkter Edge existiert → 0 Seed-Rows → Rekursion leer → immer `path_found: false`. Multi-Hop war **physisch unmöglich**.
- **Fix:** Base Case seeded jetzt ALLE Nachbarn des Startknotens:
  ```sql
  WHERE e.source_id = START OR e.target_id = START
  ```
  Plus CASE-Expression für korrekten Startknoten an Position 1.
- **Verifiziert:**
  - I/O → Laura (2 Hops: I/O → ethr → Laura) ✅
  - Laura → Drift-Projekt (3 Hops, 5 alternative Pfade gefunden) ✅
  - Dave → I/O (2 Hops: Dave → ethr → I/O) ✅

## Offene Probleme (nicht gefixt)

### P1 — L2-Dominanz
- **Problem:** 4 Meta-Insights (alle ms=0.8) dominieren JEDE ungefilterte Query. Eine Query über "Laura" liefert Ergebnisse über Memory-System-Architektur.
- **Ursache:** Sparse L2-Bestand (nur 4 Insights, alle thematisch ähnlich) + ms=0.8 Boost (×1.3)
- **Resolution:** Mehr diverse L2 Insights erstellen. Löst sich organisch.

### P2 — German Compound Words
- **Problem:** FTS 'simple' Tokenizer kann "Kontexttrennung" nicht in "Kontext" + "Trennung" zerlegen. Keyword-Kanal findet keine Compound Words.
- **Getestet:** `hybrid_search("Kontexttrennung")` → `keyword_results_count: 0`
- **Resolution:** German Stemmer (z.B. `german` Tokenizer-Config) oder n-gram Approach. Nicht trivial.

### P2 — memory_sector Klassifizierung leer
- **Problem:** Alle 131 Edges sind "semantic". Die Klassifizierungs-Regeln in Migration 022 erfassen nur `LEARNED`, `CAN_DO`, `REFLECTS`, `REALIZED` + Property-basierte Matches. Tatsächliche Relations (`BOUND_TO`, `PARTNER_OF`, `CREATED`, `UNDERSTOOD`, `VULNERABLE_TO`) sind nicht abgedeckt.
- **Konsequenz:** `sector_filter` ist für den Graph-Kanal nutzlos.

### P2 — Graph-Ergebnisse bei weight=0
- **Problem:** Graph-Resultate werden bei weight=0.0 trotzdem zurückgegeben, aber mit `rrf_score: 0`. Getestet mit `tags_filter: ["beziehung"]` — 1 Graph-Ergebnis mit Score 0.
- **Frage:** Sollen Graph-Ergebnisse bei deaktiviertem Graph-Kanal gar nicht erst geliefert werden?

### P3 — update_insight Parameter-Chaos
- **Problem:** `get_insight_by_id` nutzt `id`, `update_insight` nutzt `insight_id`. Auch `new_content` statt `content`, plus Pflichtfelder `actor` und `reason` die nicht dokumentiert sind. Fehlermeldungen zeigen nur einen fehlenden Parameter pro Versuch.
- **DX-Problem:** 4 Versuche nötig um die korrekte Signatur zu finden.

### P3 — matched_keywords Feldname irreführend
- **Problem:** Klingt nach FTS-Matches, zeigt aber nur relationale Query-Keywords ("verbunden", "connected to"). Bei Standard-Queries immer `[]`.
- **Funktioniert korrekt** — nur der Name ist irreführend.

## Nachmittags-Session: Weitere 3 Fixes (BMAD Party Mode)

*Identifiziert und gefixt in einer zweiten Session am selben Tag.*

### Fix 5: Trigram Keyword Search — `similarity()` → `word_similarity()` (SCHWERWIEGEND)
- **Dateien:** `mcp_server/tools/__init__.py` (L2 keyword ~Zeile 469, Episode keyword ~Zeile 762)
- **Problem:** `similarity(content, query)` berechnet Jaccard-Ähnlichkeit über ALLE Trigrams beider Strings. Bei 300-Wort-Dokumenten (~1500 Trigrams) gegen 3-Wort-Queries (~10 Trigrams) ergibt selbst perfekter Overlap nur ~10/1500 = 0.007 — weit unter dem Threshold von 0.15. **Der gesamte Keyword-Kanal war strukturell tot.**
- **Fix:** `word_similarity(query, content)` findet den besten Substring-Match. Threshold angepasst von 0.15 auf 0.3 (pg_trgm Default).
- **Verifiziert:** `keyword_results_count: 1` (war vorher immer 0), `episode_keyword_count: 3`

### Fix 6: `graph_query_neighbors` limit-Parameter ignoriert
- **Datei:** `mcp_server/tools/graph_query_neighbors.py` Zeile 187-192
- **Problem:** `query_neighbors()` gibt eine **Liste** zurück. Der Fix-Code prüfte `if "neighbors" in result` — das checkt ob der String `"neighbors"` ein Element in der Liste ist, nicht ob ein Dict-Key existiert. Bedingung war IMMER `False`.
- **Fix:** `isinstance(result, list) and len(result) > limit` → `result = result[:limit]`
- **Verifiziert:** `neighbor_count: 5` bei `limit=5` (war vorher 101)

### Fix 7: `get_node_by_name` — Fuzzy Fallback
- **Dateien:** `mcp_server/db/graph.py` (neue Funktion `fuzzy_search_node_by_name()`), `mcp_server/tools/get_node_by_name.py`
- **Problem:** Tool erforderte exakten String-Match. Typos oder alternative Schreibweisen lieferten `null` ohne Hinweis.
- **Fix:** Bei `not_found` automatisch `word_similarity()` Suche mit bis zu 5 Vorschlägen inkl. Similarity-Score.
- **Verifiziert:** `get_node_by_name("Drift")` → suggestions: `[{"name": "Drift-Projekt", "similarity": 1.0}]`
